Basics

01ログインと画面構成

管理者ポータルは店舗用ポータルとは別の画面です。基本メニューは「発注管理」「忘れ物一覧」「チェックシート設定」「商品設定」「棚卸表」の順に並んでいます。

ログインに使う店舗パスワードなどの情報は、ページを開いた際にサーバーから取得してブラウザ内に保存され、その保存済みの情報と照合してログイン判定を行っています。そのため、通信状況によっては同期がうまくいかず、警告バナーが表示されることがあります。表示された場合は再読み込みするか、時間をおいて再度お試しください。

Daily operation

02発注依頼への対応

各店舗パートナーから届く発注依頼を確認し、発注済みまたは否認の対応を行うのが管理者の基本業務です。

パートナーが発注依頼を送信
→
確認中
→
発注済み / 否認
→
パートナー側に通知（NEW表示）
  1. 「発注管理」タブで店舗ごとの依頼一覧を開き、確認中 の行を確認する。
  2. 内容に問題がなければ発注日・納品予定日を入力して「発注済み」にする（納品予定日は任意入力で、未定なら空欄のままでよい）。問題がある場合は「否認」にする。その上で備考欄に理由を明記すること。
  3. 「発注済み」に対応した以降、その依頼はパートナー側から取り下げることができなくなる（取り下げが可能なのは確認中の間だけ）。
  4. 発注済み・否認いずれの結果もパートナー側の画面に自動反映され、対象パートナーには通知バッジ（NEW表示）が表示される。
運用メモ 同じ商品を複数依頼したい場合、パートナー側には「追加」ボタンがないため、一度依頼を取り下げてから同じグループに品目を追加し直す運用になっています。管理者側で依頼内容を調整する際はこの点を踏まえて確認してください。
Master data

03商品設定と商品コード

「商品設定」画面では、取引先（ベンダー）ごとに商品を管理し、商品名・単価・並び順、そして商品コードを編集できます。

重要：商品コードによる商品判別 各商品には実は商品コードが割り振られています。パートナー向けの画面（発注・棚卸表など）にはコードは表示されませんが、棚卸データを外部スプレッドシートへ送信する際には、このコードも一緒に送信されています。送信先のスプレッドシート側では、商品名ではなく商品コードで商品を判別する設計になっているため、商品名を変更してもコードが同じであれば、送信先では同一商品として認識されます。逆に、コードを変更・重複させると送信先での商品判別がずれる可能性があるため、コードの編集は必ず商品設定画面から行い、既存の商品コードを他の商品に付け替えないよう注意してください。

編集できる項目

項目説明
商品コード商品を一意に識別するコード。パートナー画面には非表示。外部スプレッドシートへの送信・商品判別に使用。
商品名パートナー画面に表示される名称。
単価棚卸表の金額計算に使用（パートナー画面には非表示）。
取扱ベンダー商品が属する取引先グループ。チェックシート・棚卸表の対象品目の判定にも影響する。
並び順各行の左端のハンドルをドラッグして同一ベンダー内で並び替え可能。

店舗ごとに実際に取り扱う商品を絞り込む「店舗別取扱商品設定」も別画面にあり、ここで有効にした商品だけが、その店舗のチェックシート・棚卸表に表示されます。

Configuration

04チェックシートの設定

チェックシートは「デイリーカウント（数量記録）」と「清掃・点検（チェック項目）」の2種類に分かれており、管理者はそれぞれの対象範囲を設定できます。

デイリーカウント対象商品

デイリーカウントの対象商品は、商品設定と連動して自動的に決まります。個別に登録する必要はなく、次のルールで対象が決まります。

ベンダーデイリーカウント対象
apex / toyo / cs3すべての商品が対象
otherペーパータオルのみ対象
sales水のみ対象
tokai_snack対象外

清掃・点検項目の管理

  • チェック項目設定：清掃・点検のマスタ項目を追加・編集・削除・並び替え・カテゴリ分けする画面。
  • 店舗別チェックシート項目設定：店舗ごとに、上記マスタのうちどの項目を表示するかを選択する画面。

いずれの一覧も、カテゴリ見出しをクリックすると開閉でき、ページを開いた直後はすべて折りたたまれた状態（▶）で表示されるのが標準動作です。

Monthly closing

05棚卸表の確認

棚卸表はパートナーが月次で入力・送信する画面ですが、締め日ルールや各項目の意味を管理者も把握しておく必要があります。

締め日ルール 原則、当月末から翌月5日までに棚卸作業を行うルールです（例：7/1〜7/5の入力も6月分として扱われます＝「5日締め」）。この締め日ルールは棚卸表専用で、チェックシートのカレンダー月とは異なる点に注意してください。
項目内容
期首在庫前月の期末在庫がそのまま引き継がれる（管理者が個別入力する項目ではない）。
期末在庫パートナーが直接入力、または消費期限ごとの個数入力から自動計算される。
消費量期首在庫と期末在庫から算出される計算値。
デイリーカウントチェックシートの入力値から自動集計される参照値（棚卸表側では編集不可）。
棚卸備考月次で1つだけ入力される自由記述（商品ごとではない）。

棚卸表はパートナーの画面上、「入力」と「確認」の2画面に分かれています。入力画面では商品名・期末在庫・消費期限のみを扱い（期首在庫・消費量・デイリーカウントはここには出ません）、確認画面に進むと期首在庫・期末在庫・消費量・デイリーカウントが並んだ読み取り専用の表と、棚卸備考欄が表示されます。

消費量とデイリーカウントが一致しない商品があると、確認画面に不一致の警告が表示されます。備考が未入力のまま不一致がある場合は「棚卸完了」ボタンが押せない仕様になっているため、パートナーから「送信できない」と相談があった場合は、まず棚卸備考欄の入力を確認してください。

送信データと商品コード 「棚卸完了」を押すと、店舗の棚卸データが外部の送信先スプレッドシートに送られます。画面上は商品名しか見えませんが、送信される各行には商品コードも含まれており、送信先スプレッドシートでは商品コードをもとに商品を突き合わせています。商品名の表記だけを見て「同じ商品かどうか」を判断せず、商品が二重管理になっていないかなど疑わしい場合は商品設定画面でコードを確認してください。
Integration

06棚卸データの送信先設定

棚卸表は「棚卸完了」ボタンを押した時点で、パートナーの端末から直接ではなく、GAS（Google Apps Script）バックエンド経由で指定のスプレッドシートに書き込まれます。送信先を変更・初期設定する場合の手順です。

前提 この設定はGoogle Apps Scriptの編集権限を持つ人が行う作業で、日常的な管理者操作ではありません。パートナーポータルの画面（index.html）を変更した場合はGitHub Pagesで数分後に自動反映されますが、バックエンド（gas_backend.gs）の変更は自動反映されず、Apps Scriptエディタでの手動再デプロイが必要です。
アクセス方法 送信先の実データベース（スプレッドシート）とGASスクリプトの編集画面は、セルフカフェのGoogleアカウントでログインした状態のブラウザのブックマークバー「開発」フォルダから直接開けます。「セルフカフェ社内ポータル用データベース※セル内編集禁止」がスプレッドシート本体、「社内ポータルのGASスクリプト」がApps Scriptエディタです。スプレッドシート名にある通り、セルの内容を直接編集しないこと（棚卸完了時の自動書き込みのみを正としてください）。

仕組み

「棚卸完了」を押すと、店舗ID・期間・商品コード・商品名・期首在庫・期末在庫・消費量・デイリーカウント・単価・金額・備考などをまとめた行が、送信先スプレッドシートの inventory_log シートに書き込まれます。同じ店舗・同じ月で再送信した場合は、その月の既存データを一度削除してから新しいデータを書き込むため、重複行は残りません。商品の突き合わせは商品コードで行われます。

送信先スプレッドシートを設定・変更する手順

  1. 送信先にしたいGoogleスプレッドシートを開く（新規作成でも既存のものでも可）。URL中の https://docs.google.com/spreadsheets/d/【この部分】/edit がスプレッドシートIDなのでコピーする。
  2. ポータルのGitHubリポジトリにある gas_backend.gs を開き、INVENTORY_SHEET_ID の値を、コピーしたスプレッドシートIDに書き換える。
  3. 変更後のコードをGoogle Apps Scriptエディタに貼り付ける（コードエディタ上で直接編集している場合はそのまま保存）。
  4. Apps Scriptエディタで「デプロイ」→「デプロイを管理」→既存デプロイを選んで編集→「新しいバージョン」を選択→「デプロイ」を実行する。公開URL（/exec）は変わらないので、パートナーポータル側の設定変更は不要。
  5. 試しにいずれかの店舗で棚卸表を1件送信し、送信先スプレッドシートに inventory_log シートと行が作成されることを確認する。シートが存在しない場合はGAS側が自動で作成する。
注意 INVENTORY_SHEET_ID を書き換えたまま反映（再デプロイ）を忘れると、パートナー側は「送信完了」の表示になっていても実際には反映前のシートIDに送られてしまいます。送信先を変更した直後は必ず1件テスト送信して確認してください。また、同じ理由でIDを空欄のまま運用すると棚卸データはどこにも送信されません。

商品カテゴリ別に送信先を分けることも可能（例：お菓子だけ別シートへ）

「お菓子だけ別のスプレッドシートに送りたい」というように、商品カテゴリ単位で送信先を分岐させることもできます。各商品には商品コードとベンダー区分（仕入先カテゴリ）が設定されているため、送信時にこの区分をキーにして行を振り分け、区分ごとに別のスプレッドシートIDへ書き込む形で実現します。

棚卸完了ボタン（全商品分のデータ）
お菓子東海エリア限定お菓子（tokai_snack）
送信先A：お菓子用スプレッドシート
それ以外の商品アペックス／トーヨー／CS3／販売品／その他
送信先B：通常棚卸用スプレッドシート

現状の実装では棚卸データはすべて1つの送信先（INVENTORY_SHEET_ID）にまとめて送られる仕組みなので、上記の分岐を実際に使えるようにするには、送信先を判定・振り分けるロジックをバックエンド（gas_backend.gs）に追加する開発作業が必要です。分岐の境界は「ベンダー区分」を使うのが自然ですが、同じベンダー内にお菓子とそれ以外が混在していないかは実装前に確認しておく必要があります。
